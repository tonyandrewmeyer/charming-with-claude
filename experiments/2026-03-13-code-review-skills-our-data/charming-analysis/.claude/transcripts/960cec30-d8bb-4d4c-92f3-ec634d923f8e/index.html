Tech Stack

Component Choice Why
Backend FastAPI + uvicorn Lightweight, async, built-in OpenAPI
Database SQLite via aiosqlite Zero config, local-only
Frontend htmx (CDN) + server-rendered Jinja2 Simpler than Datastar, better docs, no build step
CSS Pico CSS (CDN) + custom stylesheet Classless defaults, looks good immediately
Code highlighting Pygments (server-side) Python snippets rendered as HTML
CLI click + httpx Maps 1:1 to API endpoints

htmx over Datastar: htmx is more mature, better documented, and the partial-swap model is a perfect fit for this (click finding -> swap detail panel). No SSE complexity needed.

Database Schema

CREATE TABLE findings (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    bug_id          TEXT NOT NULL,           -- "BUG-001"
    repo            TEXT NOT NULL,           -- "kafka-k8s-operator"
    charm_name      TEXT NOT NULL,           -- "kafka-k8s"
    round           INTEGER NOT NULL,        -- 1-4
    category        TEXT NOT NULL,           -- "Relations", "Pebble", etc.
    title           TEXT NOT NULL,           -- short description
    severity        TEXT NOT NULL CHECK(severity IN ('Critical','High','Medium','Low')),
    location        TEXT NOT NULL,           -- "src/events/provider.py:215"
    pattern         TEXT NOT NULL,           -- "AP-014" or "Novel"
    issue           TEXT NOT NULL,           -- full description
    impact          TEXT NOT NULL,
    evidence        TEXT NOT NULL,           -- code snippet (raw text)
    recommended_fix TEXT NOT NULL,
    historical_precedent TEXT NOT NULL DEFAULT '',
    review_status   TEXT NOT NULL DEFAULT 'pending'
                    CHECK(review_status IN ('pending','reviewed','false_positive')),
    reviewer_notes  TEXT NOT NULL DEFAULT '',
    reviewed_at     TEXT,
    source_file     TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(repo, bug_id)
);

CREATE TABLE confirmed_safe (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    repo            TEXT NOT NULL,
    round           INTEGER NOT NULL,
    location        TEXT NOT NULL,
    explanation     TEXT NOT NULL,
    source_file     TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE import_log (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    source_file     TEXT NOT NULL UNIQUE,
    imported_at     TEXT NOT NULL DEFAULT (datetime('now')),
    findings_count  INTEGER NOT NULL,
    safe_count      INTEGER NOT NULL DEFAULT 0
);

API Design

Browser routes return HTML; /api/* routes return JSON (used by CLI).

GET  /                              Dashboard with stats + progress
GET  /findings                      List view with filters (htmx partial: #findings-list)
GET  /findings/{id}                 Detail panel (htmx partial: #detail-panel)
PATCH /findings/{id}                Update review_status / notes (htmx re-renders row + detail)
GET  /findings/{id}/next-unreviewed Redirect to next pending finding

GET  /api/findings                  JSON list, query params: severity, status, repo, round, pattern
GET  /api/findings/{id}             JSON detail
PATCH /api/findings/{id}            JSON update
POST /api/import                    Import from markdown file path
GET  /api/stats                     Aggregate stats
GET  /api/export                    Full JSON export

Markdown Parser Strategy

The 6 skill_validation_*.md files follow a rigid structure. Parser uses regex:

  1. Extract repo/charm from header section
  2. Split on #### [BUG- to isolate findings
  3. For each finding, extract fields via regex:
  4. r'\[BUG-(\d+)\]\s+(.+?)\s+--\s+(.+?)\s+\((\w+)\)' for id/category/title/severity
  5. r'- \*\*Location\*\*:\s*(.+)' etc. for each field
  6. r'```(?:\w+)?\n(.*?)```' for code blocks
  7. Parse "Confirmed Safe" section (bullet format or paragraph format)

Round mapping (from WRITEUP.md tables): - Round 1: grafana-k8s, kafka-k8s - Round 2: discourse-k8s, hydra, opensearch, sdcore-amf

Rounds 3-4 don't have standalone validation files. Two options: 1. Create validation files for rounds 3-4 by running the skill against those repos 2. Import the narrative from WRITEUP.md as partial records

I'll build the parser for the structured files first. The CLI import command handles adding new rounds (same format).
