**How Selection Works:**

2. htmx intercepts click, reads `hx-get="/findings/{{ f.id }}"`

6. JavaScript event listener fires, marks the row as `.active`

### 6. Bug: Right Panel Shows Item 1 Instead of Selected Item

**Location of Issue**: This is likely in the **JavaScript event listener** in `findings.html` (lines 78-91).

**Potential Problems**:

1. **Event Handler Trigger Detection Issue** (Line 87):
   ```javascript
   if (trigger?.classList.contains('finding-row')) {
       trigger.classList.add('active');
   }
   ```
   - `trigger` variable is `e.detail.elt` (the element that triggered htmx)
   - If htmx was triggered by something OTHER than the finding row (e.g., a filter, button), the row won't be marked active
   - After filter changes, htmx replaces `#findings-list`, which might keep the old active class on row 1
   - When new rows are rendered, the old row 1 may still have `.active` class because `remove()` affects all elements before the detail swap

2. **Race Condition**: 
   - If filters are applied, `findings_list.html` is swapped (replacing all rows)
   - The JavaScript `htmx:afterRequest` listener removes `.active` from ALL `.finding-row` elements
   - But if a new list is inserted without a selection, the detail panel might show stale data
   - User might see details from row 1 because that was the first row rendered in the new filtered list

3. **No Active Row Tracking After Filter**:
   - When filters change, the findings list is replaced via htmx
   - The event listener doesn't know which row SHOULD be active in the new list
   - The detail panel might keep showing the old selected finding OR default to row 1

**Root Cause**: The JavaScript relies on `trigger?.classList.contains('finding-row')` to identify which row was clicked. When:
- Filters are changed → htmx target is `#findings-list`, not a finding row
- The detail panel isn't updated by the filter swap
- The old detail panel stays visible (or shows row 1 from the new list because that row happens to match the old ID)

**Evidence in Code**:
- `findings.html` line 65: Filter form has `hx-target="#findings-list"` (updates list only)
- Line 88: Active class only added if `trigger.classList.contains('finding-row')`
- No code to track which finding should be displayed when filters change

  1. Wrong event timing: htmx:afterRequest fires before the DOM swap, so reading the detail panel content got the old finding, not the new one. Fixed by splitting into htmx:afterSwap (for scroll reset) and htmx:afterSettle (for active row sync — guaranteed after DOM update).

  2. Only handled detail-panel swaps: The old handler only fired when target.id === 'detail-panel'. But Save Notes and review buttons replace the row element (target is #finding-row-{id}), stripping the .active class. The new handler fires on every htmx settle and re-syncs from the detail panel content.

  3. Filter changes left stale state: Changing filters replaced the list but didn't sync active state. Now after the list swap settles, the matching row gets .active automatically.

  4. Cascade to keyboard nav: When .active was lost, j/k navigation computed indexOf(null) = -1, then Math.min(-1+1, length) = 0 — always jumping to row 0 (item 1).

No database changes — purely frontend. Your review data is safe.
