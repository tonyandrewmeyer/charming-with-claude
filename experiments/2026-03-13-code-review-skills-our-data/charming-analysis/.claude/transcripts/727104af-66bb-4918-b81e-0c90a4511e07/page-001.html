    23→
    24→199 fixes across 61 repos. 5 high severity.
    25→
    26→### Missing can_connect() guard
    27→
    28→Pebble operations without checking container readiness crash the charm when the workload container is not yet ready or temporarily unavailable.
    29→
    30→**Before (buggy):**
    31→```python
    32→def _on_config_changed(self, event):
    33→    container = self.unit.get_container("workload")
    34→    container.push("/etc/config.yaml", config_content)
    35→    container.replan()
    36→```
    37→
    38→**After (fixed):**
    39→```python
    40→def _on_config_changed(self, event):
    41→    container = self.unit.get_container("workload")
    42→    if not container.can_connect():
    43→        event.defer()
    44→        return
    45→    container.push("/etc/config.yaml", config_content)
    46→    container.replan()
    47→```
    48→
    49→**Repos affected:** alertmanager-k8s, grafana-k8s, loki-k8s, prometheus-k8s, tempo-coordinator-k8s, discourse-k8s, synapse, many more.
    50→
    51→### Unconditional replan
    52→
    53→Calling `container.replan()` on every event causes unnecessary service restarts, potentially dropping in-flight requests.
    54→
    55→**Before (buggy):**
    56→```python
    57→def _on_config_changed(self, event):
    58→    layer = self._build_layer()
    59→    container.add_layer("app", layer, combine=True)
    60→    container.replan()  # Restarts even when nothing changed
    61→```
    62→
    63→**After (fixed):**
    64→```python
    65→def _on_config_changed(self, event):
    66→    layer = self._build_layer()
    67→    container.add_layer("app", layer, combine=True)
    68→    plan = container.get_plan()
    69→    if plan.services != layer.services:
    70→        container.replan()
    71→```
    72→
    73→### Direct plan mutation
    74→
    75→Modifying `plan.services[name].environment` in-place does not persist changes. The plan object is a snapshot.
    76→
    77→**Before (buggy):**
    78→```python
    79→plan = container.get_plan()
    80→plan.services["app"].environment["NEW_VAR"] = "value"
    81→# Change is lost -- plan is a snapshot, not a live reference
    82→```
    83→
    84→**After (fixed):**
    85→```python
    86→new_layer = Layer({
    87→    "services": {"app": {"environment": {"NEW_VAR": "value"}}}
    88→})
    89→container.add_layer("app", new_layer, combine=True)
    90→container.replan()
    91→```
    92→
    93→### Missing ChangeError handling on restart
    94→
    95→`container.restart()` can raise `ChangeError` (e.g., HTTP 429 rate limit from Pebble). Unhandled, this crashes the charm.
    96→
    97→**Before (buggy):**
    98→```python
    99→container.restart("workload")
   100→```
   101→
   102→**After (fixed):**
   103→```python
   104→try:
   105→    container.restart("workload")
   106→except ChangeError as e:
   107→    logger.warning("Failed to restart: %s", e)
   108→    event.defer()
   109→```

   111→### None or non-string values in Pebble environment dicts
   112→
   113→Pebble rejects None values and may mishandle non-string values in service environment dictionaries. Python `bool` values become `"True"`/`"False"` (capital) which may not match what Go expects (`"true"`/`"false"` lowercase).

   116→```python
   117→env = {
   118→    "DB_HOST": self._get_db_host(),  # May return None

   125→```python

   127→    "DB_HOST": self._get_db_host(),

   136→## Relations and Events
   137→
   138→59 fixes across 39 repos. 4 high severity.
   139→
   140→### relation_departed vs relation_broken confusion
   141→
   142→`relation_departed` fires when a single unit leaves; `relation_broken` fires when the entire relation is removed. Using departed for full cleanup causes premature resource removal while other units remain.
   143→
   144→**Before (buggy):**
   145→```python
   146→def _on_db_relation_departed(self, event):
   147→    # Wrong: removes DB config even when other DB units remain
   148→    self._clear_database_config()
   149→    self.unit.status = BlockedStatus("Database removed")
   150→```
   151→
   152→**After (fixed):**
   153→```python
   154→def _on_db_relation_broken(self, event):
   155→    self._clear_database_config()
   156→    self.unit.status = BlockedStatus("Database relation removed")
   157→```
   158→
   159→### Missing relation_broken handler
   160→
   161→When `relation_joined`/`relation_changed` sets up resources, a missing `relation_broken` handler leaves stale configuration.
   162→
   163→**How to detect:** Search for relations with `relation_joined` or `relation_changed` handlers but no corresponding `relation_broken`.
   164→
   165→### event.relation.app is None
   166→
   167→In some events (e.g., during relation teardown), `event.relation.app` can be None. Accessing `.data[event.relation.app]` raises KeyError.
   168→
   169→**Before (buggy):**
   170→```python
   171→def _on_relation_changed(self, event):
   172→    remote_data = event.relation.data[event.relation.app]
   173→```
   174→
   175→**After (fixed):**
   176→```python
   177→def _on_relation_changed(self, event):
   178→    if event.relation.app is None:
   179→        event.defer()
   180→        return
   181→    remote_data = event.relation.data[event.relation.app]
   182→```
   183→
   184→### Relation data overwrite in loop
   185→
   186→When iterating over multiple relations, overwriting (instead of appending) loses data from all but the last relation.
   187→
   188→**Before (buggy):**
   189→```python
   190→for relation in self.model.relations["database"]:
   191→    connection_info = self._get_connection(relation)
   192→    # Overwrites previous iteration's result
   193→```
   194→
   195→**After (fixed):**
   196→```python
   197→connections = []
   198→for relation in self.model.relations["database"]:
   199→    connections.append(self._get_connection(relation))
   200→```

   310→## Status Management
   311→
   312→243 fixes across 79 repos. 14 high severity.
   313→
   314→### Status set in wrong handler
   315→
   316→Setting `ActiveStatus` in one handler when another subsystem is not ready. Status should reflect the aggregate state of all subsystems.
   317→
   318→**Before (buggy):**
   319→```python
   320→def _on_config_changed(self, event):
   321→    self._apply_config()
   322→    self.unit.status = ActiveStatus()
   323→    # But database might not be related yet!
   324→```
   325→
   326→**After (fixed):**
   327→```python
   328→def _on_config_changed(self, event):
   329→    self._apply_config()
   330→    self._update_status()
   331→
   332→def _update_status(self):
   333→    if not self._database_ready():
   334→        self.unit.status = BlockedStatus("Waiting for database relation")
   335→    elif not self._container_ready():
   336→        self.unit.status = WaitingStatus("Waiting for container")
   337→    else:
   338→        self.unit.status = ActiveStatus()
   339→```
   340→
   341→### Missing status on early return
   342→
   343→Event handlers that `return` early without setting status leave the charm in the previous status, which may be stale.
   344→
   345→**Before (buggy):**
   346→```python
   347→def _on_relation_changed(self, event):
   348→    if not self.unit.is_leader():
   349→        return  # Status unchanged -- may show "Active" when actually waiting
   350→```
   351→
   352→**After (fixed):**
   353→```python
   354→def _on_relation_changed(self, event):
   355→    if not self.unit.is_leader():
   356→        self.unit.status = ActiveStatus()  # Or appropriate status for non-leader
   357→        return
   358→```
   359→
   360→### Blocked vs Waiting confusion
   361→
   362→`BlockedStatus` means user action required. `WaitingStatus` means the charm is waiting for an automatic event (relation, Pebble ready). Using the wrong one misleads operators.
   363→
   364→### Contradictory unit vs app status
   365→
   366→Setting `BlockedStatus` on the unit but `WaitingStatus` on the app (or vice versa) for the same condition sends mixed signals to operators. `juju status` shows both, and they should agree.
   367→
   368→**Before (buggy):**
   369→```python
   370→event.add_status(BlockedStatus("Waiting for MetalLB"))
   371→self.app.status = WaitingStatus("Waiting for MetalLB")  # Contradicts unit!
   372→```
   373→
   374→**After (fixed):**
   375→```python
   376→event.add_status(BlockedStatus("Waiting for MetalLB to be enabled"))
   377→self.app.status = BlockedStatus("Waiting for MetalLB to be enabled")
   378→```
   379→
   380→**Repos affected:** sdcore-amf-operator.
