Confirmed Bugs

BUG-1: Mutable Default Argument (Python Anti-Pattern)

  • File: /home/ubuntu/charming-analysis/repos/mongodb-k8s-operator/tests/integration/helpers.py, line 70
  • Severity: Low (test code only, but can cause subtle test failures)
  • Code: async def get_app_name(ops_test: OpsTest, test_deployments: List[str] = []) -> str:
  • Problem: The default argument test_deployments: List[str] = [] uses a mutable default. If any caller were to mutate the list (e.g., test_deployments.append(...) inside the function body or via a reference), the mutation would persist across calls. This is a well-known Python footgun.
  • Impact: In practice, get_app_name only reads from test_deployments (app in test_deployments), so the mutable default is not actively mutated. However, this is still a code smell that linters flag, and any future modification could introduce a latent bug.
  • Fix: Change to test_deployments: List[str] | None = None and add if test_deployments is None: test_deployments = [] at the start.

BUG-2: Return Type Annotation Mismatch -- get_app_name returns None but declared as -> str

  • File: /home/ubuntu/charming-analysis/repos/mongodb-k8s-operator/tests/integration/helpers.py, line 70 and 91
  • Severity: Low (test code only)
  • Code: Function signature says -> str but line 91 does return None.
  • Problem: The return type annotation is incorrect. The function can return None when no matching app is found, but is annotated as returning str. Callers like test_build_and_deploy (line 31 of test_charm.py) check if app_name:, which handles the None case, but get_direct_mongo_client (line 213) calls get_app_name(ops_test, APP_NAME) -- note this passes APP_NAME (a string) as the test_deployments parameter, which means it's passing a string where a List[str] is expected. The in operator on a string checks for substrings, not list membership, so app in "mongodb-k8s" would match any app name that is a substring of "mongodb-k8s" (e.g., "mongo", "db", "k8s"), which could cause incorrect filtering.
  • Impact: Medium for test reliability. The incorrect type passed to test_deployments means substring matching instead of list-element matching, potentially causing wrong apps to be filtered out or, conversely, failing to filter apps that should be excluded.
  • Fix: Change annotation to -> str | None and fix the call at line 213 to pass [APP_NAME] instead of APP_NAME.

BUG-3: get_direct_mongo_client passes string where list expected

Bug 1: Certificate Cleanup Uses Wrong Filename Slicing (SEVERITY: HIGH)

File: /home/ubuntu/charming-analysis/repos/traefik-k8s-operator/src/traefik.py Lines: 199, 224, 226, 237

The certificate cleanup logic uses hardcoded string slices ([:5], [:4], [:18]) to extract what should be the hostname (dict key) from filenames, but these slice lengths do not correctly strip the file extensions.

  • Line 199: path.name[:5] on a file like "local-config.cert" yields "local", not "local-config".
  • Line 224: path.name[:5] on "10.152.183.5.cert" yields "10.15", not "10.152.183.5".
  • Line 226: path.name[:4] on "local-config.key" yields "loca", not "local-config".
  • Line 237: path.name[:18] on "local-config.traefik-charm.crt" yields "local-config.traef", not "local-config".

The correct approach would be to strip the extension (e.g., path.name.removesuffix(".cert") or os.path.splitext(path.name)[0] for .cert/.key, and path.name.removesuffix(".traefik-charm.crt") for CA certs).

Impact: Valid certificates that should be retained are incorrectly deleted, and certificates that should be removed may be retained. This causes TLS breakage after any cert rotation or update event. The bug affects both the local filesystem cleanup (line 199) and the Pebble container cleanup (lines 224, 226, 237).

Reachable: Yes -- called from _update_cert_configs() which is called on every _configure(), pebble-ready, and config-changed event.
