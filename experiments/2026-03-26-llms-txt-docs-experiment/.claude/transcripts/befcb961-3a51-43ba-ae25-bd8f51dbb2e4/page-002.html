## Q10: How do you deploy a charm and wait for it to reach active/idle using jubilant?

**Source files:** `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/tutorial/getting-started/index.html.md` and `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/reference/jubilant/index.html.md`

The documentation shows this in both the tutorial and API reference. Complete test code:

```python
import pytest
import jubilant


@pytest.fixture(scope='module')
def juju():
    with jubilant.temp_model() as juju:
        yield juju


def test_deploy(juju: jubilant.Juju):
    juju.deploy('snappass-test')
    juju.wait(jubilant.all_active)

    # Or wait for just 'snappass-test' to be active (ignoring other apps):
    juju.wait(lambda status: jubilant.all_active(status, 'snappass-test'))
```

Key details from the docs:
- `jubilant.Juju()` creates the Juju wrapper. The constructor accepts an optional `model` parameter and a `wait_timeout` (default 180 seconds).
- `jubilant.temp_model()` is a context manager that creates a randomly-named model (format `jubilant-abcd1234`) and destroys it on exit.
- `juju.deploy()` takes a charm name (from Charmhub) or a path to a `.charm` file (must start with `/` or `.`).
- `juju.wait(ready, error=..., timeout=..., successes=3)` polls `juju status` repeatedly. The `ready` callable must return `True` for `successes` times in a row (default 3) before `wait` returns.
- `jubilant.all_active` checks that **all** apps and **all** their units have "active" status. You can pass specific app names to filter.
- `jubilant.all_agents_idle` is the separate helper to check that all unit agents are "idle" (this checks the Juju agent status, not the workload status).
- To wait for both active **and** idle, you would combine them:
  ```python
  juju.wait(lambda status: jubilant.all_active(status) and jubilant.all_agents_idle(status))
  ```
- You can also pass `error=jubilant.any_error` to raise a `WaitError` if any app/unit enters error state during the wait.

## Q11: How do you add a relation between two applications and verify data was exchanged, using jubilant?

**Source files:** `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/reference/jubilant/index.html.md`, `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/tutorial/getting-started/index.html.md`, `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/reference/statustypes/index.html.md`

To add a relation (integration), use `juju.integrate()`:

```python
def test_active_apps(juju: jubilant.Juju):
    for app in ['blog', 'mysql', 'redis']:
        juju.deploy(app)
    juju.integrate('blog', 'mysql')
    juju.integrate('blog', 'redis')
    juju.wait(
        lambda status: jubilant.all_active(status, 'blog', 'mysql', 'redis'),
    )
```

Key details from the docs:
- `juju.integrate(app1, app2)` creates a relation. The order of app1/app2 is not significant. Each can be in format `<application>[:<endpoint>]` -- the endpoint is only required when there is more than one possible integration between the two apps.
- The `via` parameter is available for cross-model relations (CIDR notation).
- `juju.remove_relation(app1, app2)` removes a relation.

**Verifying data was exchanged:** Jubilant does **not** have a dedicated `show-unit` or `relation-get` method in its API. The documented approaches for verification are:

1. **Check relation existence via status:** After integrating, `status.apps['myapp'].relations` is a `dict[str, list[AppStatusRelation]]` where each `AppStatusRelation` has `related_app`, `interface`, and `scope` fields. This confirms the relation exists.

2. **Check application reaches active status:** The typical pattern is to wait for all apps to become active after integrating, which implicitly confirms the relation was established and data exchanged (since charms typically go active only after receiving required relation data).

3. **Use `juju.cli()` as fallback:** For reading raw relation data, you can fall back to the CLI:
   ```python
   import json
   stdout = juju.cli('show-unit', 'myapp/0', '--format=json')
   unit_data = json.loads(stdout)
   # Inspect relation-info in the parsed output
   ```

4. **Use actions:** Many charms expose actions (like `get-password`) to retrieve data that was set via relations, which is the documented pattern for verifying exchanged data (see Q12).

## Q12: How do you run a Juju action on a unit and check its results using jubilant?

**Source file:** `/home/ubuntu/llms-txt-experiment/docs-output/jubilant/reference/jubilant/index.html.md`

Use `juju.run()`:

```python
juju = jubilant.Juju()
result = juju.run('mysql/0', 'get-password')
assert result.results['username'] == 'USER0'
```

Key details from the docs:
- **Signature:** `juju.run(unit, action, params=None, wait=None) -> Task`
  - `unit`: unit name like `mysql/0` or `mysql/leader`
  - `action`: action name string
  - `params`: optional `Mapping[str, Any]` of named parameters to pass to the action
  - `wait`: optional timeout in seconds (Juju default is 60s)
- **Return type is `Task`** with these attributes:
  - `task.results`: `dict[str, Any]` -- the results provided by the charm (excludes special keys `return-code`, `stdout`, `stderr` which are separate attributes)
  - `task.status`: one of `'aborted'`, `'cancelled'`, `'completed'`, `'error'`, `'failed'`
  - `task.success`: `bool` property -- whether the action was successful
  - `task.message`: failure message if the charm provided one when failing the action
  - `task.return_code`: `int` (default 0)
  - `task.stdout` / `task.stderr`: strings
  - `task.log`: `list[str]` of messages logged by the action hook
  - `task.id`: task ID string
- **Error handling:** `juju.run()` raises `TaskError` if the action failed, `ValueError` if the action or unit does not exist, and `TimeoutError` if `wait` was specified and exceeded.
- There is also `task.raise_on_failure()` to manually raise `TaskError` if the task was not successful.

A more complete test example with params and error handling:

```python
def test_action(juju: jubilant.Juju):
    juju.deploy('mysql')
    juju.wait(jubilant.all_active)

    result = juju.run('mysql/leader', 'get-password')
    assert result.success
    assert result.status == 'completed'
    assert 'username' in result.results
```

## Q18: Declaring a relation endpoint in `charmcraft.yaml`

**Source:** `/home/ubuntu/llms-txt-experiment/docs-output/charmcraft/reference/files/charmcraft-yaml-file/index.html.md`, lines 1042-1170

Relation endpoints are declared under three possible top-level keys based on the endpoint role: `peers`, `provides`, or `requires`.

### Full schema

```yaml
<endpoint role>:  # One of 'peers', 'provides', or 'requires'
  <endpoint name>:
    # (Required) The interface schema that this relation conforms to.
    # Cannot be "juju". Cannot begin with "juju-".
    # Must only contain characters a-z and - and cannot start with -.
    interface: <endpoint interface name>

    # (Optional) Maximum number of supported connections to this relation endpoint.
    # Integer.
    limit: <n>

    # (Optional) Defines if the relation is required. Boolean.
    # Default: false
    # Not enforced by Juju, but used by other tools and should always be included.
    optional: true | false

    # (Optional) The scope of the relation. String.
    # Possible values: "global", "container"
    # Default: "global"
    # Container-scoped integrations restrict reporting to a single principal
    # unit to a single subordinate, and vice versa.
    scope: global | container
```

### Supported fields per endpoint

| Field | Status | Type | Notes |
|---|---|---|---|
| `interface` | **Required** | String | The interface name; cannot be `juju` or start with `juju-`; only `a-z` and `-` |
| `limit` | Optional | Integer | Maximum number of connections |
| `optional` | Optional | Boolean | Whether the relation is required; defaults to `false` |
| `scope` | Optional | String | `global` (default) or `container`; subordinate charms must have at least one `requires` with `container` scope |

### Example featuring all three keys

```yaml
peers:
  friend:
    interface: life
    limit: 150
    optional: true
    scope: container
provides:
  self:
    interface: identity
requires:
  parent:
    interface: birth
    limit: 2
    optional: false
    scope: global
```

## Q19: Resource declarations for OCI images

**Source:** `/home/ubuntu/llms-txt-experiment/docs-output/charmcraft/reference/files/charmcraft-yaml-file/index.html.md`, lines 1174-1213

### Key rule

Kubernetes charms **must** declare an `oci-image` resource for each container they define in the `containers` mapping.

### Structure

```yaml
resources:
  <resource-name>:
    type: <file> or <oci-image>
    description: <string description of the resource>
    filename: <path to resource if it is a file>
```

For OCI images specifically, the `filename` field is not used -- only `type: oci-image` and an optional `description` are needed.

### OCI image example

```yaml
resources:
  super-app-image:
    type: oci-image
    description: OCI image for the Super App (hub.docker.com/_/super-app)
```

### Linking a resource to a container

The resource name declared under `resources` must match the `resource` field in the corresponding `containers` entry:

```yaml
containers:
  super-app:
    resource: super-app-image
    mounts:
    - storage: logs
      location: /logs

resources:
  super-app-image:
    type: oci-image
    description: OCI image for the Super App (hub.docker.com/_/super-app)
```

### File resource example (for comparison)

```yaml
resources:
```

## Q6: Complete Pebble layer YAML with HTTP health check, custom environment variables, and a log target

Based on the layer specification at `/home/ubuntu/llms-txt-experiment/docs-output/pebble/reference/layer-specification/index.html.md`, a complete layer combining all three features:

```yaml
summary: Example layer with service, health check, env vars, and log target

services:
    my-webapp:
        override: replace
        command: /usr/bin/my-webapp --port 8080
        startup: enabled
        environment:
            APP_ENV: production
            DB_HOST: localhost
            DB_PORT: "5432"
            LOG_LEVEL: info
        on-check-failure:
            http-check: restart

checks:
    http-check:
        override: replace
        level: ready
        period: 10s
        timeout: 3s
        threshold: 3
        http:
            url: http://localhost:8080/health
            headers:
                Accept: application/json

log-targets:
    my-loki:
        override: replace
        type: loki
        location: http://loki-host:3100/loki/api/v1/push
        services: [my-webapp]
        labels:
            env: production
```

Key facts from the docs:
- **Services**: `override` is required (merge or replace). `command` is required in the combined layer. `environment` is a map of key/value pairs. `startup` defaults to "disabled".
- **Checks**: Only one of `http`, `tcp`, or `exec` may be specified. An HTTP check succeeds if a GET to the URL returns a 2xx status code. `period` defaults to 10s, `timeout` to 3s, `threshold` to 3. The `services` list determines which services' logs are forwarded; use `all` for all services. Pebble automatically adds a `pebble_service` label. Custom `labels` values may contain `$ENV_VARS` that get substituted per-service.
- **Tying checks to services**: Use `on-check-failure` in the service definition, mapping check names to actions (`restart`, `shutdown`, `success-shutdown`, or `ignore`).

## Q7: Pebble notices -- types, mechanics, and ops integration

Source: `/home/ubuntu/llms-txt-experiment/docs-output/pebble/reference/notices/index.html.md`

**What notices are**: Notices are a subsystem that allows introspection of events occurring in the Pebble server, plus recording custom client events. They are saved to disk (persist across restarts) and expire after a notice-defined interval.

**Identity and deduplication**: Each notice is uniquely identified by its **user ID**, **type**, and **key** combination. When a notice with the same (user-id, type, key) occurs again, the occurrence count is incremented rather than creating a new notice.

**Visibility**: Each notice is either public or has a specific user ID. Public notices can be viewed by any user. User-specific notices can only be viewed by that user or by an admin.

**Repeat behaviour**: A "repeat" happens when a notice with the same user ID, type, and key occurs and either there is no "repeat after" duration, or it happens after the "repeat after" interval has elapsed. This prevents high-frequency notices from flooding consumers.

**Data**: A notice records optional string key-value pair data from the last occurrence, plus timestamps for first-occurred, last-occurred, and last-repeated.

**Notice types** (three types exist):

1. **`change-update`**: Recorded whenever a change is first spawned or its status is updated. The key is the change ID, and the notice's data includes the change `kind`.
2. **`custom`**: A custom client notice reported via `pebble notify`. The key and data are user-provided. The key must be in the format `example.com/path` to ensure well-namespaced keys.
3. **`warning`**: Pebble warnings implemented as notices. The key is the human-readable warning message.

**How a charm observes notices via ops**: The Pebble documentation at `/home/ubuntu/llms-txt-experiment/docs-output/pebble/reference/api/index.html.md` references the Python client in the Ops library. The Python client for the Pebble API is part of `ops.pebble` (documented at `https://documentation.ubuntu.com/ops/latest/reference/pebble/#ops-pebble`). In the Juju/ops context, when Pebble emits a `custom` notice, Juju detects it and dispatches a `pebble_custom_notice` event on the charm. The charm can observe this with `self.on.pebble_custom_notice` (or the container-specific variant). The notice's type, key, and data are available on the event object. ## Q8: Pebble change/task model and checking long-running operation completion

Sources: `/home/ubuntu/llms-txt-experiment/docs-output/pebble/reference/changes-and-tasks/index.html.md`, `/home/ubuntu/llms-txt-experiment/docs-output/pebble/reference/api/index.html.md`, `/home/ubuntu/llms-txt-experiment/docs-output/pebble/openapi.yaml`

**The model**: When Pebble performs a potentially invasive or long-running operation (starting/stopping a service, etc.), it records a **change** object containing one or more **tasks**. The state is persisted in a JSON file at `$PEBBLE/.pebble.state` (unless `PEBBLE_PERSIST` is set to "never", in which case state is kept in memory only).

**Change structure** (from OpenAPI examples):
- `id`: unique change identifier
- `kind`: type of operation (e.g., `autostart`, `start`, `stop`, `perform-check`, `recover-check`)
- `summary`: human-readable description
- `status`: one of `Done`, `Error`, `Abort`, or in-progress states
- `ready`: boolean indicating whether the change has completed
- `spawn-time` / `ready-time`: timestamps
- `tasks`: array of task objects, each with its own `id`, `kind`, `summary`, `status`, `progress` (with `done`/`total` counters), and timestamps

**How to check if a long-running operation has completed**:

1. **CLI**: `pebble changes` lists recent changes with their status. `pebble tasks <change-id>` shows the individual tasks within a change.

2. **API -- polling**: `GET /v1/changes/{id}` returns the change with its current status and `ready` boolean.

3. **API -- blocking wait**: `GET /v1/changes/{id}/wait` blocks until the change finishes. Accepts an optional `timeout` query parameter (a duration string). If specified, it waits until the change is ready or the timeout occurs, whichever comes first. If not specified or zero, it waits indefinitely.

4. **API -- filtering**: `GET /v1/changes?select=in-progress` filters to only in-progress changes. The `select` parameter accepts `all`, `in-progress`, or `ready`. An optional `for` parameter filters by service name.

5. **API -- abort**: `POST /v1/changes/{id}` with `{"action": "abort"}` aborts a change.

6. **Go client**: Use `pebble.WaitChange(changeID, &client.WaitChangeOptions{})` to block until completion.

7. **Python client (ops)**: The Python client methods like `client.stop_services()` automatically wait for the change to finish, according to the docs: "Python client also waits for change to finish".

8. **Notices**: A `change-update` notice is recorded whenever a change is first spawned or its status is updated. The key is the change ID, and the data includes the change `kind`. Clients can long-poll `GET /v1/notices?types=change-update&timeout=30s` to be notified of change status updates.

## Q4: What's the difference between `ActionEvent.fail()` and raising an exception in an action handler?

**Sources:**
- `/home/ubuntu/llms-txt-experiment/docs-output/operator/reference/ops/index.html.md`, lines 64-69
- `/home/ubuntu/llms-txt-experiment/docs-output/operator/howto/manage-actions/index.html.md`, lines 99-103
- `/home/ubuntu/llms-txt-experiment/docs-output/operator/howto/write-and-structure-charm-code/index.html.md`, lines 291-309

**`ActionEvent.fail(message: str = '')`:**
- Reports that the action has failed with an optional message explaining why.
- **Critically, `fail()` does not interrupt code execution.** The docs explicitly state: "Note that `fail()` doesn't interrupt code, so is typically followed by a `return`." This means after calling `fail()`, the handler continues running unless you explicitly return.
- This is the **controlled, graceful** way to report action failure. The failure message is sent back to the administrator who invoked the action.
- The action's status is set to "failed" with the provided message visible to the user.

**Raising an exception:**
- An unhandled exception in any charm handler (including an action handler) causes the charm to **crash and enter the error state**.
- The docs say: "Unexpected/unrecoverable error: the charm should enter the error state. Do this by raising an appropriate exception in the charm code. Note that the unit status will only show an `error` status, and the user will need to use the Juju log to get details of the problem."
- This puts the entire unit into error state (not just the action), and the user must use `juju resolve` to recover. The error details are only in the Juju log, not in the action output.

**In summary:** `event.fail()` is a graceful, controlled failure that reports a message to the user and lets you clean up; raising an exception is an uncontrolled crash that puts the entire unit into error state and requires manual intervention.

## Q4: What's the difference between `ActionEvent.fail()` and raising an exception in an action handler?

**`ActionEvent.fail(message: str = '')`:**
- Reports that the action has failed with an optional message.
- **Does not interrupt code execution** — typically followed by a `return`.
- The failure message is sent back to the administrator who invoked the action.
- The action's status is set to "failed" with the message visible to the user.
- This is the **controlled, graceful** way to report action failure.

**Raising an exception:**
- An unhandled exception causes the charm to **crash and enter the error state**.
- Puts the entire unit into error state (not just the action).
- The user must use `juju resolve` to recover.
- Error details are only in the Juju log, not in the action output.

**Summary:** `event.fail()` is a graceful, controlled failure that reports a message to the user and lets you clean up; raising an exception is an uncontrolled crash that puts the entire unit into error state and requires manual intervention.

**Source:** ops API reference (`ActionEvent.fail`) and ops how-to guide ("Write and structure charm code").

## Q5: How do you use `ops.CollectStatusEvent` to set charm status? Show a complete example.

`CollectStatusEvent` is triggered at the end of every hook. The framework automatically picks the highest-priority status from all statuses added via `add_status()`.

**Priority order** (highest to lowest): `error` > `blocked` > `maintenance` > `waiting` > `active`

It is an error to call `add_status()` with `ErrorStatus` or `UnknownStatus`.

**Complete example:**
```python
class MyCharm(ops.CharmBase):
    def __init__(self, framework: ops.Framework):
        super().__init__(framework)
        framework.observe(self.on.collect_app_status, self._on_collect_app_status)
        framework.observe(self.on.collect_unit_status, self._on_collect_unit_status)

    def _on_collect_app_status(self, event: ops.CollectStatusEvent):
        # Triggered for the leader unit only.
        num_degraded = ...  # inspect peer databags
        if num_degraded:
            event.add_status(ops.ActiveStatus(f'degraded units: {num_degraded}'))
            return
        event.add_status(ops.ActiveStatus())

    def _on_collect_unit_status(self, event: ops.CollectStatusEvent):
        # Triggered for each unit.
        if not self.model.config.get('port'):
            event.add_status(ops.BlockedStatus('please set "port" config'))
            return
        event.add_status(ops.ActiveStatus())
```

**Key behaviours:**
- `collect_unit_status` fires for every unit after every hook.
- `collect_app_status` fires only on the leader unit.
- Multiple `add_status()` calls are allowed; ops picks the highest priority.
- Multiple handlers can observe the same event (e.g., component-based pattern).

## Q7: How do Pebble notices work? What types are there and how does a charm observe them via ops?

**What notices are:** A subsystem for introspection of events in Pebble. Notices are saved to disk (persist across restarts) and expire after a defined interval.

**Identity and deduplication:** Each notice is uniquely identified by (user ID, type, key). Repeated occurrences increment a counter rather than creating new notices.

**Notice types:**

1. **`change-update`** — Recorded when a change is spawned or its status is updated. Key = change ID. Data includes the change `kind`.
2. **`custom`** — Client-reported via `pebble notify`. Key must be in `example.com/path` format. Key and data are user-provided.
3. **`warning`** — Pebble warnings. Key = human-readable warning message.

**How a charm observes notices via ops:**
- When Pebble emits a `custom` notice, Juju detects it and dispatches a `pebble_custom_notice` event.
- The charm observes it with `self.on.<container>_pebble_custom_notice`.
- The event object provides the notice's type, key, and data.

**API:** `GET /v1/notices` supports filtering by `types`, `keys`, `user-id`, `after` timestamp, and a `timeout` parameter for long-polling.

**CLI:** `pebble notices` (list), `pebble notice <id>` (get one), `pebble notify <key>` (record custom notice).

**Source:** Pebble reference — "Notices".

## Q8: What is the Pebble change/task model? How do you check if a long-running operation has completed?

**The model:** When Pebble performs a potentially long-running operation (starting/stopping services, etc.), it records a **change** containing one or more **tasks**. State is persisted to `$PEBBLE/.pebble.state`.

**Change structure:**
- `id`, `kind`, `summary`, `status`, `ready` (bool), `spawn-time`, `ready-time`
- `tasks`: array of task objects, each with `id`, `kind`, `summary`, `status`, `progress` (with `done`/`total` counters)

**How to check completion:**

1. **CLI:** `pebble changes` lists recent changes. `pebble tasks <change-id>` shows tasks within a change.
2. **API polling:** `GET /v1/changes/{id}` — check the `ready` boolean.
3. **API blocking wait:** `GET /v1/changes/{id}/wait` — blocks until change finishes. Accepts optional `timeout` query parameter.
4. **API filtering:** `GET /v1/changes?select=in-progress` — filter by state (`all`, `in-progress`, `ready`).
5. **Notices:** A `change-update` notice fires when a change is spawned or updated. Long-poll with `GET /v1/notices?types=change-update&timeout=30s`.
6. **Python client (ops):** Methods like `client.stop_services()` automatically wait for the change to finish.

**Source:** Pebble reference — "Changes and tasks".

## Q20: What bases/platforms does charmcraft support and how do you specify multi-base builds?

**Supported bases:** `ubuntu@22.04`, `ubuntu@24.04`, `ubuntu@24.10`, `ubuntu@25.04`, `almalinux@9`. Also `ubuntu@devel` as `build-base`.

**Supported architectures:** `amd64`, `arm64`, `armhf`, `ppc64el`, `s390x`.

**Single-base (standard):**
```yaml
base: ubuntu@24.04
platforms:
  amd64:
```

**Multiple architectures:**
```yaml
base: ubuntu@24.04
platforms:
  amd64:
  arm64:
```

**Cross-compilation:**
```yaml
base: ubuntu@24.04
platforms:
  riscv64-cross:
    build-on: [amd64]
    build-for: [riscv64]
```

**Multi-base builds:** The `base` top-level key is **not** used. Instead, the base is embedded in each platform entry:

```yaml
platforms:
  ubuntu@22.04:amd64:
  ubuntu@24.04:amd64:
```

Or in long form:
```yaml
platforms:
  ubuntu-22.04-amd64:
    build-on: [ubuntu@22.04:amd64]
    build-for: [ubuntu@22.04:amd64]
  ubuntu-24.04-amd64:
    build-on: [ubuntu@24.04:amd64]
    build-for: [ubuntu@24.04:amd64]
```

Both produce two `.charm` files.

**Architecture-independent:**
```yaml
base: ubuntu@24.04
platforms:
  all:
    build-on: [amd64]
    build-for: [all]
```

**Source:** charmcraft reference — "Platforms" and "charmcraft.yaml file".

## S3: Charm that provides data over a custom relation interface

```python
#!/usr/bin/env python3

import ops


class ProviderCharm(ops.CharmBase):
    def __init__(self, framework: ops.Framework):
        super().__init__(framework)
        framework.observe(
            self.on.my_service_relation_joined,
            self._on_relation_joined,
        )

    def _on_relation_joined(self, event: ops.RelationJoinedEvent):
        # Set the endpoint key in the relation data to this unit's FQDN
        # Unit databag is writable by the unit itself
        event.relation.data[self.unit]['endpoint'] = self._get_fqdn()

    def _get_fqdn(self) -> str:
        import socket
        return socket.getfqdn()


if __name__ == '__main__':
    ops.main(ProviderCharm)
```

```yaml
provides:

```

- Must observe `relation_joined` (or `relation_changed`) — `relation_joined` is most appropriate for "when a remote app joins".

     3→# `prometheus_scrape/v0`
     4→
     5→## Usage
     6→
     7→This relation interface describes the expected behavior of any charm claiming to be able to provide or require Prometheus Scrape Endpoints.
     8→
     9→In most cases, this will be accomplished using the [prometheus_scrape library](https://charmhub.io/prometheus-k8s/libraries/prometheus_scrape), although charm developers are free to provide alternative libraries as long as they fulfill the behavioral and schematic requirements described in this document.
    10→
    11→## Direction
    12→
    13→As all Juju relations, the `prometheus_scrape` interface consists of a provider and a requirer. One of these, in this case the `Provider`, will be expected to stand up one or more scrape compatible metrics endpoint where the `Requirer` will be able to scrape the `Provider`s metrics.
    14→
    15→## Behavior
    16→
    17→Both the Requirer and the provider need to adhere to a certain set of criterias to be considered compatible with the interface.
    18→
    19→### Provider
    20→
    21→- Is expected to provide one or more Prometheus scrape jobs in the relation data bag.
    22→- Is expected to respect the metrics topology set by the requirer.
    23→- Is expected to provide alert rules over the relation data bag.
    24→- Is expected to add any wanted topology labels to all metrics sent to the provider.
    25→- Is expected to provide any wanted topology label matchers as labels on every alert rule in the relation data bag.
    26→- Is expected to be able to expose both single alert rules and alert rule groups over the relation data bag.
    27→- Is expected to send alert rules where the contain expressions lacks Juju topology label selectors.
    28→
    29→### Requirer
    30→
    31→- Is expected to be able to scrape Prometheus metrics from a metrics endpoint
    32→- Is expected to be able to ingest alert rules exposed over the relation data bag.
    33→- Is expected to fetch the target configuration from the relation data bag.
    34→- Is expected to inject alert rule topology labels as label matchers in alert rule expressions.
    35→- Is expected not to inject juju_unit as a label matcher by default, but to honor it if hard-coded by the user.
    36→- Is expected to be able to ingest both single alert rules and alert rule groups provided over the relation data bag.
    37→
    38→## Relation Data
    39→
    40→[[Pydantic Schema]](https://github.com/canonical/charmlibs/blob/main/interfaces/prometheus_scrape/interface/v0/./schema.py)
    41→
    42→### Provider
    43→
    44→- Exposes all scrape jobs the requirer should scrape metrics through. Should be placed in the **application** databag.
    45→- Exposes the unit address of each unit to be scraped, along with the unit name. Additionally, it includes an optional path provided by the unit for Prometheus scraping. This path is present when the provider charm is backed by an Ingress or a Proxy. Should be placed in the **unit** databag of each scrapable unit.

# pathops Library Overview

The `pathops` charm library provides a `pathlib`-like interface for working with both local and remote container filesystem paths in Juju charms.

## Key Components

**PathProtocol**: A protocol defining common methods for both local and container paths, enabling type-annotated code that works across Kubernetes and machine charms.

**ContainerPath**: Implements remote path operations for workload containers using the Pebble file API. It only supports absolute paths and raises `RelativePathError` for relative paths.

**LocalPath**: Extends `pathlib.PosixPath` with enhanced file-creation arguments like `mode`, `user`, and `group` for consistency with `ContainerPath`.

## Notable Features

Both implementations support standard path operations: `read_text()`, `write_bytes()`, `mkdir()`, `iterdir()`, and `glob()`. Container operations may raise `PebbleConnectionError` if the workload container is unreachable.

The `ensure_contents()` helper function works with both path types, verifying file existence, contents, permissions, and ownership while returning whether modifications occurred.

File write methods accept optional `mode`, `user`, and `group` parameters to set permissions and ownership on creation or modification.

- [`LocalPath`](#pathops.LocalPath): the concrete implementation of the interface for local paths, which includes
  both machine charms and the charm container of Kubernetes charms. Inherits from
  [`pathlib.PosixPath`](https://docs.python.org/3/library/pathlib.html#pathlib.PosixPath) and extends the signature of some methods.
- Top-level helper functions such as [`ensure_contents`](#pathops.ensure_contents), which operate on both container
  and local paths.

[`ContainerPath`](#pathops.ContainerPath) methods that interact with the remote filesystem will raise a
[`PebbleConnectionError`](#pathops.PebbleConnectionError) if the workload container isn’t reachable.
Methods designed to work with both local and container paths may handle this error internally,
or they may leave handling connection errors to the caller, documenting this if so.

def build_run_plan(
    questions: list[dict],
    conditions: list[str],
    models: list[str],
    runs: int,
    seed: int = 42,
) -> list[dict]:
    """Build a randomised run plan."""
    plan = []
    for run_number in range(1, runs + 1):
        for question in questions:
            for condition in conditions:
                for model in models:
                    session_id = (
                        f"{question['id']}_{condition}_{model}_run{run_number}"
                    )
                    plan.append({
                        "question": question,
                        "condition": condition,
                        "model": model,
                        "run_number": run_number,
                        "session_id": session_id,
                    })

    # Randomise order within each run
    rng = random.Random(seed)
    # Group by run, shuffle within each run
    by_run: dict[int, list] = {}
    for item in plan:
        by_run.setdefault(item["run_number"], []).append(item)
    result = []
    for run_num in sorted(by_run):
        items = by_run[run_num]
        rng.shuffle(items)
        result.extend(items)

    return result

def weighted_score(scores: dict, is_synthesis: bool) -> float:
    """Calculate weighted score for a response."""
    weights = SYNTH_WEIGHTS if is_synthesis else IR_WEIGHTS
    max_score = SYNTH_MAX if is_synthesis else IR_MAX
    total = 0
    for dim, weight in weights.items():
        total += scores.get(dim, 0) * weight
    return total / max_score * 100  # Normalise to percentage


def analyse_by_condition(results: list[dict]) -> dict:
