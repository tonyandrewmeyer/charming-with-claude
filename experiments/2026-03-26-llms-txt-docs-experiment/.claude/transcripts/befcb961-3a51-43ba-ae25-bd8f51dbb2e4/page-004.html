Assistant2026-03-26T08:32:21.681Z

Good question. I'd lean towards yes, add them, for a few reasons:

  1. Realism — if Canonical were to adopt llms.txt across their docs, Juju would be included. The agent already tries to fetch /juju/ (we saw it in the logs). Having real content there instead of a 404 is more representative.

  2. The llms.txt index value — one of the things we're testing is whether the agent navigates to the right repo. If it can fetch /juju/llms.txt and see "this covers the Juju CLI and controller, not charm development" it might learn to go to ops/pebble/charmcraft instead. That's a genuine benefit of llms.txt that we'd miss by not serving it.

  3. Synthesis questions (S1-S3) — these combine knowledge. An agent might look at Juju docs for concepts like relations, models, or actions even when the code-level answer lives in ops. Having the Juju docs available tests whether llms.txt helps or hurts when the agent has more doc surface to navigate.

  4. It shouldn't bias the scores — we're scoring against gold standards from ops/pebble/jubilant/charmlibs/charmcraft. Extra docs that don't help don't hurt the score, and if they do help, that's a genuine finding.

The downside is Juju's docs are likely large, so llms-full.txt might be hefty, and it's another build to get working. The Juju repo is canonical/juju (Go project, Sphinx docs). Do you have a fork of that one too, or should we just build from upstream since we're not changing anything (only adding sphinx-llm to conf.py locally)?
