## Sphinx "Show Source" / "View Source" Feature

### 1. Configuration options

There are three relevant Sphinx configuration options:

- **`html_copy_source`** (bool): Controls whether Sphinx copies the original source files (`.rst`, `.md`, etc.) into the build output. Default: **`True`** (enabled by default in vanilla Sphinx).
- **`html_show_sourcelink`** (bool): Controls whether a "Show Source" link appears in the sidebar of each page. Default: **`True`** (enabled by default in vanilla Sphinx).
- **`html_sourcelink_suffix`** (str): The suffix appended to source filenames in the output. Default: `".txt"` -- so `index.rst` becomes `index.rst.txt`.

### 2. Where Sphinx puts the source files

When `html_copy_source = True`, Sphinx copies the original source files into:

```
<build-output>/_sources/
```

The directory structure mirrors the documentation source tree. For example:
- `docs/tutorial/getting-started.md` becomes `_sources/tutorial/getting-started.md.txt`
- `docs/reference/api.rst` becomes `_sources/reference/api.rst.txt`

### 3. Format

The files are copies of the **original source** (`.rst` or `.md`) with the suffix `.txt` appended (controlled by `html_sourcelink_suffix`). So:
- `.rst` files become `.rst.txt`
- `.md` files become `.md.txt`

The content is the raw markup, not rendered HTML.

# Research Findings: Markdown via HTTP Content Negotiation

## 1. Convention/Standard for Requesting Markdown via Accept Headers

Yes, there is a growing convention. The pattern is straightforward HTTP content negotiation:

```
Accept: text/markdown, text/html, */*
```

This follows the same mechanism used for decades to negotiate between HTML and JSON. The client states its preference, the server decides what to serve. Several AI coding tools have already adopted this convention (see point 5 below).

## 2. MIME Type: Is `text/markdown` Official?

**Yes, it is an official IANA-registered media type**, standardised in **RFC 7763** (March 2016).

- **Type:** `text/markdown`
- **Optional parameters:** `charset` (defaults to UTF-8), `variant` (e.g., `CommonMark`, `GFM`, `MultiMarkdown`)
- **Companion RFC 7764** defines a registry of markdown variants

Example: `Content-Type: text/markdown; charset=UTF-8; variant=GFM`

This is not informal -- it is fully standardised.

## 3. Tools/Servers/CDNs Supporting This Approach

The ecosystem has grown significantly:

**CDN/Edge level:**
- **Cloudflare "Markdown for Agents"** -- a production feature (Pro/Business/Enterprise plans) that performs real-time HTML-to-markdown conversion at the edge when it detects `Accept: text/markdown`. Returns an `x-markdown-tokens` response header with estimated token count. Origin responses up to 2MB are supported.

**Documentation platforms:**
- **Mintlify** -- supports both llms.txt and content negotiation
- **Fern** -- supports LLM-friendly markdown serving
- **Vercel** -- confirmed support

**Web servers (via configuration):**
- **Caddy** -- straightforward matcher rules and conditional rewrites
- **Cloudflare Workers** -- custom JS to inspect Accept headers and route between HTML/markdown directories
- **Nginx** -- configurable but requires manual setup
- **Static Web Server** (Rust-based) -- has a dedicated "Markdown Content Negotiation" feature

**Performance data:**

- [Cloudflare's announcement](https://blog.cloudflare.com/markdown-for-agents/) -- "Markdown has quickly become the lingua franca for agents and AI systems"

**The emerging consensus is that they are complementary, not competing:**
- **llms.txt** serves as a discovery/index mechanism (like `robots.txt` for crawlers)
- **Content negotiation** serves as the per-page delivery mechanism
- Together: llms.txt tells agents what exists; Accept headers let agents request it in the right format
