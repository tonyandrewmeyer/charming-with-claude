## 1. What is llms.txt?

**llms.txt** is a proposal for a standardised file placed at a website's root path (`/llms.txt`) to provide language models with concise, curated information about a site's content. It was authored by **Jeremy Howard** (co-founder of fast.ai) and published on **3 September 2024**.

The core problem it addresses: LLM context windows cannot accommodate entire websites, and converting complex HTML to usable text is imprecise. By providing a single, expert-curated markdown file, sites can give LLMs structured access to the most important information.

It is **not** an official IETF or W3C standard -- it remains a community-driven proposal.

## 2. The Specification Format

The file uses a strict markdown structure, in this order:

1. **H1 heading** (required) -- the project or site name
2. **Blockquote** (optional) -- a brief summary with key information
3. **General content** (optional) -- paragraphs, lists (no headings) with detailed project information
4. **H2-delimited sections** (optional) -- containing "file lists" as markdown links

File list entries follow this format:
```
- [Link title](url): Optional descriptive notes
```

A special **"Optional"** H2 section designates URLs that can be skipped when a shorter context is needed.

The spec also recommends providing clean markdown versions of pages by appending `.md` to the original URL (e.g., `/docs/api.html.md`).

## 3. What is llms-full.txt?

**llms-full.txt** is a companion convention (not formally part of the original spec, but widely adopted alongside it) that contains the **complete text content** of a site's documentation in a single markdown file.

The key difference:

| File | Purpose | Content |
|------|---------|---------|
| `llms.txt` | Navigation map | Links with one-sentence descriptions; helps AI find specific pages |
| `llms-full.txt` | Complete reference | Full text of all documentation pages in one file |

There are also derivative formats from the FastHTML ecosystem:
- `llms-ctx.txt` -- expanded content without embedded URLs
- `llms-ctx-full.txt` -- expanded content with full URLs

These are generated via the `llms_txt2ctx` CLI tool.

According to Mintlify's data, **LLMs access llms-full.txt even more frequently than the original llms.txt**, suggesting the complete content format is more practically useful.

## 4. Intended Use Cases

The specification emphasises **inference-time** use -- when users are actively seeking information via an LLM -- rather than for training purposes. Key use cases include:

- Developer documentation and API references
- Organisational structure for businesses
- Regulatory/legal content simplification
- E-commerce product and policy information
- Educational course content
- Personal portfolios and CVs

The idea is that when a user asks an AI assistant a question about (say) a framework's API, the LLM can fetch `/llms.txt` to understand the documentation structure and then retrieve the relevant pages.

## 5. Adoption

Adoption numbers vary significantly depending on the source:

- **BuiltWith** tracked ~844,000 websites with llms.txt as of October 2025
- **SE Ranking** found a **10.13% adoption rate** across 300,000 analysed domains
- **NerdyData** reported only **951 domains** as of July 2025

Notable adopters and endorsements:
- **Anthropic** specifically asked Mintlify to implement llms.txt and llms-full.txt for Claude's documentation
- **Google** included llms.txt in their Agents-to-Agents (A2A) protocol
- **Vercel** reports 10% of signups now come via ChatGPT through Generative Engine Optimisation efforts
- Documentation platforms like **Mintlify**, **Fern**, **GitBook**, and **Redocly** support generating llms.txt
- **Windsurf** highlighted that llms.txt saves time and tokens when agents do not need to parse complex HTML

## 6. Evidence of Effectiveness

The evidence is **mixed to negative** for current impact, but with some forward-looking signals:

**Against effectiveness (current):**
- SE Ranking's analysis of 300,000 domains found **no correlation** between llms.txt presence and AI citations. Removing the llms.txt variable actually *improved* their predictive model's accuracy.
- A crawl audit found **zero visits** from GPTBot, ClaudeBot, PerplexityBot, or Google-Extended to llms.txt pages over a multi-month period (mid-August to late October 2025).
- Search Engine Land reported 8 out of 9 sites saw **no measurable change** in traffic after implementation.
- No major LLM provider has publicly confirmed their crawlers consistently read or follow llms.txt.

**For effectiveness (emerging):**
- **Profound** (a GEO tracking company) collected data showing Microsoft, OpenAI, and others are actively crawling and indexing both llms.txt and llms-full.txt files.
- Anthropic's direct request for Mintlify to implement the format suggests at least one major AI lab sees value in it.
- One case study (Springs Apps) reported a 20% increase in search visibility and 15% improvement in accurate AI answers, though this is unverified by independent sources.

**Bottom line:** llms.txt is low-effort to implement and carries no technical risk, but there is currently no strong evidence that it meaningfully affects how AI systems cite or use your content. Its value may be more prospective than present -- a bet on future AI agent architectures that explicitly consume structured site metadata.

## Research Summary: sphinx-llm and Alternatives

### 1. What does sphinx-llm do?

**sphinx-llm** (current version 0.3.0, released 6 February 2026) is a collection of Sphinx extensions for working with LLMs, developed under the NVIDIA GitHub organisation (primary author: Jacob Tomlinson). It serves two purposes:

- **Enabling LLMs/agents to consume your docs** -- produces additional build output (markdown files) following the [llms.txt standard](https://llmstxt.org/) so that LLMs and AI agents can efficiently ingest your documentation.
- **Leveraging LLMs to generate content** -- uses LLMs (via Ollama) during the Sphinx build process to dynamically generate content that gets baked into the documentation.

It provides two extensions:

1. **`sphinx_llm.txt`** -- Generates llms.txt-standard output files.
2. **`sphinx_llm.docref`** -- Adds a directive that auto-summarises and cross-references other documentation pages using a local LLM.

### 2. How does it integrate with Sphinx?

You add the extensions to your `conf.py`:

```python
extensions = [
    "sphinx_llm.txt",      # for llms.txt generation
    "sphinx_llm.docref",   # for LLM-generated cross-references (optional)
]
```

The `sphinx_llm.txt` extension hooks into the Sphinx build process. It:
1. Runs the normal HTML build.
2. Runs a parallel markdown builder alongside it.
3. Merges both outputs together.
4. Creates an `llms.txt` index file.
5. Concatenates all markdown into a single `llms-full.txt` file.

It only works with HTML builders (`html` and `dirhtml`).

The `sphinx_llm.docref` extension requires Ollama running locally and uses a content-hashing system to avoid regenerating summaries unnecessarily.

### 3. Does it generate llms.txt files? Markdown files? Both?

**Both.** It generates:

- **`llms.txt`** -- An index/summary file listing the documentation structure (in markdown).
- **`llms-full.txt`** -- A single concatenated markdown file containing all documentation.
- **Individual `.html.md` files** -- A markdown version of each individual HTML page, placed alongside the HTML output. This means every page on your site has a corresponding `.md` version accessible by appending `.md` to the URL.

All output is in **Markdown** format.

### 4. How is it configured?

Install with:
```bash
pip install sphinx-llm          # core (llms.txt generation)
pip install sphinx-llm[gen]     # adds LLM content generation (docref)
```

Configuration options in `conf.py`:

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `llms_txt_description` | str | From pyproject.toml | Override project description in llms.txt |
| `llms_txt_build_parallel` | bool | `True` | Build markdown files in parallel with HTML |
| `llms_txt_suffix_mode` | str | `"auto"` | Controls markdown file naming: `"auto"`, `"file-suffix"`, `"url-suffix"`, or `"replace"` |
| `llms_txt_full_build` | bool | `True` | Whether to generate the llms-full.txt file |

The `docref` directive is used inline in RST files:
```rst
.. docref:: page-name
   :hash: 31ec12a54205539af3cde39b254ec766
   :model: llama3.2:3b

   Auto-generated summary text here.
```

### 5. Alternatives for generating LLM-friendly docs from Sphinx

#### sphinx-llms-txt (by Jared Dillard)

- **PyPI:** [sphinx-llms-txt](https://pypi.org/project/sphinx-llms-txt/)
- **GitHub:** [jdillard/sphinx-llms-txt](https://github.com/jdillard/sphinx-llms-txt)
- **Docs:** [sphinx-llms-txt.readthedocs.io](https://sphinx-llms-txt.readthedocs.io/)
- **Current version:** 0.7.1 (MIT licence)
- **Also available on conda-forge**

This is a more focused, mature alternative that only generates `llms.txt` and `llms-full.txt` (it does not generate per-page markdown files or use LLMs for content generation). Key differences:

- Outputs `llms-full.txt` in **reStructuredText** (not Markdown), preserving the original source format.
- Richer configuration options including: custom filenames (`llms_txt_filename`, `llms_txt_full_filename`), size limits (`llms_txt_full_max_size`), content exclusion via glob patterns (`llms_txt_exclude`), source code inclusion (`llms_txt_code_files`), URI templates, and custom directive handling.
- Does **not** generate individual per-page markdown files.
- Parallel-build safe.

There is an [open issue on the Sphinx project](https://github.com/sphinx-doc/sphinx/issues/13268) requesting native `llms-full.txt` support. The Sphinx maintainers have indicated they are unlikely to add this to core without "significant user demand / funding" and recommend using community extensions instead.

   # Add sphinx_llm.txt to extensions in conf.py

   ```

2. **Serve locally via nginx** with path-based routing:
   ```

   # localhost/ops/latest/llms.txt
   # localhost/ops/latest/llms-full.txt
   # localhost/ops/latest/reference/api.html.md
   # localhost/pebble/latest/...
   # etc.
   ```

3. **DNS override** via `/etc/hosts`:
   ```
   127.0.0.1 documentation.ubuntu.com
   ```
   This redirects all doc fetches to the local server. The real docs (HTML) are also served locally so the agent has access to everything it would normally have, plus the new LLM-friendly files.

### Alternative: MCP/Tool-Level Override

Instead of DNS tricks, we could:
- Provide a custom MCP server or tool that serves docs
- Configure Claude Code to use it for doc lookups
- Simpler but less realistic (the agent "knows" it's using a special tool)

### Alternative: Custom Domain

- Less realistic but avoids breaking real doc access during testing
