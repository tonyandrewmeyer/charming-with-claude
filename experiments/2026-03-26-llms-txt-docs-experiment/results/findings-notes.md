# Findings Notes for Write-Up

Observations collected during the experiment that should be included in the final write-up.

## URL Pattern Hallucination

### The `/en/` prefix

The agent frequently uses `/en/latest/` or `/en/stable/` in URLs when fetching from `documentation.ubuntu.com`. This was a URL pattern used historically but is not part of the current URL structure for any of the charm ecosystem repos.

**Data:** 29% of all doc fetches (211/726) included `/en/`. The breakdown by condition is stark:
- **Condition C** (bare agent + llms.txt hint): 50% of fetches use `/en/`, in 68% of sessions
- **Condition D** (instructions + llms.txt): only 4% of fetches use `/en/`, in 2 sessions

The instructions in condition D apparently give the agent enough context to use correct URLs. Without instructions, the agent falls back to training-data URL patterns.

The most common hallucinated pattern was `/juju/en/latest/` (82 hits). Even `/charmlibs/en/latest/` appeared 14 times despite charmlibs never having had a version prefix.

**Implication for adoption:** If Canonical adopts llms.txt, they should ensure old URL patterns redirect properly. Without our nginx rewrite rules, condition C would have hit 404s on 50% of its fetches, dramatically reducing the value of llms.txt. URL stability and redirects are a prerequisite for llms.txt to be useful.

### Old domains: juju.is and ops.readthedocs.io

The charm ecosystem docs have migrated domains over time:
- **juju.is** → documentation.ubuntu.com/juju/
- **ops.readthedocs.io** → documentation.ubuntu.com/ops/

The agent's training data likely includes content from both old domains. Since our experiment only intercepts `documentation.ubuntu.com` via `/etc/hosts`, any fetches to `juju.is` or `ops.readthedocs.io` go to the real internet (which redirects to the current domain). This means:

1. Condition A/B agents might find docs via old domains and still get content (via redirects on the real internet).
2. Condition C/D agents that try old domains would bypass our local server entirely, missing the llms.txt files.

## llms-full.txt Is Barely Used

The agent almost never fetches `llms-full.txt` — only 6% of condition C fetches, 1% of condition D. It strongly prefers fetching `llms.txt` (the index) and then following links to individual pages.

This suggests the per-page `.html.md` files generated by sphinx-llm are more valuable than the full concatenation. If a project can only do one thing, generating `llms.txt` as a navigation index is more impactful than `llms-full.txt`.

## Condition D: Instructions May Interfere With Doc Exploration

Condition D (instructions + llms.txt) scores 75.0% — below condition C's 78.5%. This is counterintuitive: more context should help.

Possible explanation from the data: condition D fetches llms.txt only 15% of the time (vs C's 36%), and uses HTML 71% of the time (vs C's 36%). The instructions seem to make the agent more confident in its existing knowledge, causing it to fetch docs less and rely more on what it already "knows" from the instructions. When the instructions are incomplete or slightly wrong, the agent doesn't cross-check against the docs.

This is worth investigating further — does condition D do fewer total fetches than C? (Yes: 4.9 vs 5.0 fetches per session, with 109KB vs 255KB bytes fetched.) The instructions reduce doc exploration, which is helpful for efficiency but may reduce accuracy when the instructions don't cover the specific question.

## Hallucination Rate Is Slightly Higher With llms.txt

Conditions C and D show 4.5-4.8% hallucination rate vs 0-1.5% for A/B. The sample is small (3 hallucinations each out of ~65 sessions), so this may be noise. But a possible explanation: when the agent fetches docs and finds relevant-looking content, it may over-confidently interpolate details that aren't actually in the fetched page.

Worth spot-checking: are the hallucinations in C/D cases where the agent fetched a relevant page but then invented details that weren't on it?

## The Agent Gravitates to Juju Docs First

In condition C, the agent's first fetch is `/juju/en/latest/llms.txt` **27 times out of 66 sessions** — even for questions about ops, pebble, or charmcraft. The agent strongly associates `documentation.ubuntu.com` with Juju (the most prominent project on the domain in training data).

In condition D, the first fetch is distributed across the correct repos because the instructions name them explicitly.

**Implication:** A generic "docs support llms.txt" hint isn't enough — the agent needs to know which sub-sites exist. The llms.txt at `/juju/llms.txt` should ideally link to sibling projects (ops, pebble, etc.) to help with discovery.

## Synthesis Tasks: Instructions Can Suppress Code Output

For S3 (write a provider charm), conditions B and D score dramatically lower (32-33%) than A and C (75-77%). Investigation reveals the cause: **the agent with instructions often describes what the code would do rather than writing it**. It outputs `charmcraft.yaml`, explains the approach, references file line numbers, but never actually provides the `src/charm.py` implementation.

Conditions A and C (no instructions) just write the code directly, which is what the question asks for.

This pattern is also visible in S2 (pebble-ready handler) and S1 to a lesser extent. The CLAUDE.md instructions appear to trigger a more cautious, tool-oriented workflow (plan, describe, reference files) rather than direct code output. In interactive use this is fine (the agent would use Write/Edit tools), but in our `-p` mode the agent can't actually write files — it can only output text.

**Implication for the experiment:** The synthesis task scores for B/D may understate the value of instructions. In real interactive use, the agent would use tools to actually write the files. Our non-interactive `-p` mode penalises the instructed agent's preferred workflow.

**Implication for instructions:** Consider adding guidance like "When asked to write code, output the complete code directly rather than describing what you would write" for non-interactive contexts.
